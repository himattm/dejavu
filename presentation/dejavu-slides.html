The Promise of Compose

  • Declarative UI — describe what, not how
  • Smart recomposition — only recomposes what changed
  • The runtime handles it ™
0:30 — Set up the ideal world. "Compose promises smart recomposition. You describe your UI, the runtime figures out the minimal work. That's the contract. But it's implicit — nowhere in your code does it say which composables should or should not recompose." ~1 minute.

The Reality

  • Unstable parameters cause silent recompositions
  • Lambda captures break skip optimizations
  • State hoisting mistakes propagate recompositions
  • No test catches this. It just gets slower.
1:30 — "But here's what actually happens. You add a lambda, forget to remember it, and suddenly half your tree recomposes every frame. The worst part? No test fails. Your CI is green. Your app is just... slower. And you don't know until a user complains or you open Layout Inspector." ~1.5 minutes.

What if recomposition was testable?


@get:Rule
val composeTestRule = createRecompositionTrackingRule<CounterActivity>()

@Test
fun counterTitle_staysStable() {
    // Click increment multiple times
    repeat(3) {
        composeTestRule.onNodeWithTag("inc_button").performClick()
    }

    // CounterTitle should never recompose (no inputs changed)
    composeTestRule.onNodeWithTag("counter_title").assertStable()
}

This is Dejavu.

3:00 — "What if you could write this? One rule. Standard Compose test APIs. And then assert that a composable did not recompose. This is Dejavu." ~1 minute.

Part I

Inception

4:00 — Transition beat. "Let me take you back to where this started."

Inspiration: Square's Workflow

  • Square's Workflow library tracks render passes
  • Every render is explicit and countable
  • You can assert: "this screen rendered exactly N times"
  • Why can't Compose do the same?
4:15 — "I was working with Square's Workflow library, which has this concept of render tracking. Every time your screen renders, you know about it, you can count it, you can assert on it. And I kept thinking — Compose has recomposition, which is conceptually the same thing, but there's zero testing infrastructure for it." ~1.5 minutes.

The Implicit Contract

What you write

  "Title",
  modifier = Modifier.testTag("title")
)

What you assume

  • This doesn't recompose when sibling state changes
  • But nothing enforces that
  • And the Compose compiler can change skipping behavior between versions

5:45 — "When you write this Text composable with a hardcoded string, you're implicitly assuming it will be skipped during recomposition. But that's just an assumption. There's no contract, no test, no assertion." ~1.5 minutes.

August 23, 2025 — Initial release

The Counter Example


@Composable
fun CounterScreen() {
    var count by remember { mutableIntStateOf(0) }
    Column(modifier = Modifier.testTag("counter_screen")) {
        CounterTitle()
        CounterValue(count)
        IncButton { count++ }
        ResetButton { count = 0 }
    }
}

@Composable fun CounterTitle() {
    Text("Dejavu Counter", modifier = Modifier.testTag("counter_title"))
}

@Composable fun CounterValue(value: Int) {
    Text("Value: $value", modifier = Modifier.testTag("counter_value"))
}

counter_title — never recompose (no params change)
counter_value — must recompose (reads count)

9:15 — "Here's the simplest possible example. A counter. The title text is hardcoded — it should never recompose. The value text reads count — it must recompose. With Dejavu, you can assert both sides of that contract." ~1.5 minutes.

Under the Hood: 5 Steps

  1. Intercept — Composer.setTracer(DejavuTracer)
  2. Map — Two-pass walk of CompositionData Group trees
  3. Count — First call = composition (skip), subsequent = recomposition (count)
  4. Causality — Snapshot observer + dirty bits decode
  5. Report — Structured multi-section error messages
13:15 — "Under the hood, there are five steps. Let me walk through each one." ~1 minute.

Counting & Causality

Counting

  • First traceEventStart(key) = initial composition → skip
  • Subsequent calls with same key = recomposition → increment counter
  • Per-function counting via qualified name

Causality

  • Snapshot.registerApplyObserver detects state changes
  • Dirty bits reveal which parameter slots triggered recomposition
  • Records type of changed state (Int, String, List...)

  Recomposition timeline:
    #1 at +0ms — param slots changed: [1] | parent: ProductListScreen

  Possible cause:
    1 state change(s) of type Int
    Parameter/parent change detected (dirty bits set)
16:15 — "Counting is straightforward — skip the first call, count the rest. Causality tracking is where it gets interesting — we observe Snapshot state changes and inspect dirty bits to tell you exactly why a recomposition happened." ~2 minutes.

v0.1.0 What Was Wrong

  • No per-instance tracking
    5 Star composables with star_0...star_4 all mapped to the same Star() function count
  • Logging was global, not per-tag
    Hard to debug which specific instance recomposed
  • Hardcoded framework filter
    ~40 entries like Column, Box, Button — broke when Google shipped new components
  • Fragile across Compose BOM versions
    InspectableValue API format changed silently between BOMs
20:15 — "But v0.1.0 had real problems. The biggest: no per-instance tracking. If you had a list of items, all instances of the same composable were lumped together. And the framework filter was a hardcoded list that broke every time Google shipped a new Compose version." ~1.5 minutes.

v0.1.1 Per-Instance Tracking

March 3, 2026


@Composable
fun Star(index: Int, isFilled: Boolean, onClick: () -> Unit) {
    Text(
        text = if (isFilled) "★" else "☆",
        fontSize = 40.sp,
        color = if (isFilled) Color(0xFFFFD700) else Color.Gray,
        modifier = Modifier
            .testTag("star_$index")  // dynamic tag per instance
            .size(48.dp)
            .pointerInput(Unit) { detectTapGestures { onClick() } }
    )
}
  

@Test
fun multiInstance_unchangedInstancesReportZero() {
    composeTestRule.onNodeWithTag("set_rating_3_btn").performClick()

    // Stars that changed show recomposition
    composeTestRule.onNodeWithTag("star_0").assertRecompositions(atLeast = 1)

    // Stars that didn't change stay stable
    composeTestRule.onNodeWithTag("star_3").assertStable()
    composeTestRule.onNodeWithTag("star_4").assertStable()
}
21:45 — "v0.1.1 solved the per-instance problem. Each testTag is tracked independently. So if you have a star rating with 5 stars, you can assert that star_0 recomposed because it became filled, while star_3 stayed stable." ~2 minutes.

Key Concepts

Design decisions that matter

25:45 — Transition. "Let me highlight the design decisions that shaped Dejavu."

Design Decision: No Custom Modifiers

Other approaches


// Requires production code change
Text(
  "Title",
  modifier = Modifier
    .trackRecomposition("title")
    .testTag("title")
)

Dejavu


// Standard testTag — already there
Text(
  "Title",
  modifier = Modifier
    .testTag("title")
)

Your production code doesn't change. Your test tags do double duty.

26:00 — "The most important design decision: no custom modifiers. Other approaches require you to add special modifiers to production code. Dejavu uses the testTag you already have. Zero production code changes." ~1.5 minutes.

Two-Pass Mapping Algorithm

Pass 1 — Group Tree Walk
Traverse CompositionData.compositionGroups recursively.
For each Group: record source location, group key, modifier info.

Pass 2 — Semantics Correlation
For each Group with a testTag in its modifier chain,
find the nearest ancestor that represents a user composable function call.

Result
Map: "counter_title" → demo.app.ui.CounterTitle (Counter.kt:28)

[IMAGE: Group tree with testTag nodes highlighted, arrows pointing up to their parent composable functions]
27:30 — "The two-pass mapping is the core algorithm. First pass walks the internal Group tree. Second pass connects testTag nodes to their enclosing composable. This is what lets us say 'counter_title belongs to CounterTitle at line 28'." ~1.5 minutes.

Composed Modifiers: The Trap


// In older Compose versions, testTag is implemented as:
fun Modifier.testTag(tag: String): Modifier = composed {
    semantics { this.testTag = tag }
}
// composed {} creates its own recomposition scope!
  • The modifier itself can recompose even when the composable doesn't
  • Naive counting would count modifier recompositions, not composable ones
  • Dejavu filters these out — tracks at the composable function level
29:00 — "Here's a subtlety that took a while to get right. In older Compose versions, testTag is a composed modifier with its own recomposition scope. If you naively count recompositions, you count the modifier recomposing, not the composable. Dejavu handles this by tracking at the composable function level." ~1.5 minutes.

Frame-Aligned Tracking


// Runtime.kt — Choreographer frame loop
Choreographer.getInstance().postFrameCallback {
    // After each frame:
    // 1. Capture latest CompositionData snapshots
    // 2. Refresh tag-to-function mappings
    // 3. Update recomposition counts
    DejavuTracer.buildTagMappingFromFrameLoop(latestSnapshots)
}
  • Tracking aligns with Android's frame cadence
  • Assertions read stable counts after the frame completes
  • This is why assertions are deterministic
30:30 — "The lifecycle management uses Android's Choreographer to align tracking with composition frames. This is what makes assertions deterministic — you perform an action, the frame completes, and then your assertion reads stable counts." ~1.5 minutes.

Assertion Flexibility


// Exact count

Real-World Pattern


@Test
fun scrollMessages_channelNameBarStaysStable() {
    // Trigger a UI interaction
    composeTestRule.onNodeWithTag("ConversationTestTag")
        .performTouchInput { swipeUp() }
    composeTestRule.waitForIdle()

    // Assert stable components didn't recompose
    composeTestRule.onNodeWithTag("ChannelNameBar")
        .assertStable()

    // Assert dynamic components did recompose
    composeTestRule.onNodeWithTag("MessageList")
        .assertRecompositions(atLeast = 1)
}

Part V

AI Safety

Keeping AI agents informed when they update your UI

36:15 — "There's one more angle I want to cover that I think is increasingly important."

AI Agents & Compose

  • AI coding agents increasingly write and modify Compose UI code
  • They optimize for "does it compile and render correctly?"
  • An AI can easily introduce a lambda capture or unstable parameter
  • The AI has no signal that it broke performance

// AI "helpfully" refactors — introduces unstable lambda
ProductItem(
    product = product,
    onSelect = { viewModel.select(product.id) }  // new lambda every recomposition!
)
36:30 — "AI coding agents are writing more and more UI code. But they have no understanding of recomposition. An AI will happily refactor your composable in a way that compiles, passes all functional tests, but causes 10x more recompositions. And it has no signal that anything went wrong." ~1.5 minutes.

Dejavu as AI Guardrail

  • Recomposition tests in CI catch AI-introduced regressions
  • When Dejavu fails, the structured error tells the AI exactly what went wrong
  • The AI can read the error and fix the issue autonomously

Structured errors aren't just for humans.

38:00 — "Dejavu tests in your CI become a guardrail. When an AI agent makes a change and runs the tests, a Dejavu failure gives it a structured error message with composable name, file location, expected vs actual, and a possible cause. That's enough for the AI to understand and fix the problem without human intervention." ~1.5 minutes.
